硝烟中的Scrum和XP

作者：[瑞] Henrik Kniberg

  • 0. 前言

    • 一个大牛因为生病卧床，决定记录下自己在过去几年中所学到的知识以及所积累的经验而写出的一份长文总结。

      参考链接：敏捷软件开发之Scrum(扫盲篇)

      + 本书能为你找到适合自己的一套方法提供参考，进而帮助你做到更熟悉自己的产品，帮助你更好地了解自己团队的生产率等，这就是本书的价值所在。
      另外这本书介绍的是作者实施Scrum过程的经验总结，注重的是实践而非理论研究，所以这也是本书的独到之处！

      + Scrum 和 XP 是敏捷开发的具体方式，Scrum(争球)和XP(极限编程)都要求团队在每一次迭代的结尾都完成一些可以交付的工作片段。

     

    思考：

    • @本书所讲述的scrum/sprint方法，我觉得不仅仅适用于软件开发过程，而且应该具有更广泛的适用场景！它相当于是一套以人为核心的，迭代的，循环渐进式的完成某个大目标的方法论！
      比如：你有一个长期的目标想要完成(这就相当于是你的一个Product)，首先你可以列出你的Product-Backlog，然后将其划分成n个sprint(将这个大任务分解)，列出每个sprint的Sprint-Backlog，然后再逐步完成你的sprint，并在每个阶段结束时拿出可见的成果进行总结和Review，相信按照这一套做下来最终你一定能够得到你想要的那个最初的Product！

     

    全书自我总结：

    • - Scrum 是一种完成大目标的方法论！
      它不仅适用于软件开发过程，它也可以适用于任何你想要去完成的你觉得是一个大目标的东西 (比如：你的人生目标，你的长期目标，你的短期目标等等)。

       

      - 其实说白了，Scrum 方法就是为了给你实现 整个大目标的过程中添加“信息反馈机制”！
      但是这一点真的很重要，为什么我们在实现一个大目标的过程中很容易忘了初衷而选择放弃？就是因为实现整个目标不是一天两天就能做到的，它需要一个长期的持续不断的坚持和努力，而在这个努力的过程中，努力的效果又不会特别明显，所以往往大多数人都因为发现努力了一段时间没有成效而选择放弃！其实，是真的没有成效吗？只是我们看不见，其实这平静的海面下已经暗流涌动！
      所以，在这个过程中为了让你能够坚持下去，一个人为的看得见的“信息反馈”就显得格外重要，让你知道我的努力即使是一天也是有成效的，这样你就会愿意花时间继续去做着这件事，渐渐的你也就坚持了下来！

       

      - 本书详细的给我们介绍了如何使用 scrum 方法
      从一开始的【如何制定大目标任务清单】到【如何选取和制定一个sprint冲刺阶段的目标任务清单】再到【怎么进行任务，怎么开会，怎么做演示，怎么做回顾总结】最后谈到【如何对团队进行有效管理】。这些东西相当于是一个已经运用这一套方法很多年经验的人，给那些想要运用这个方法帮助提升自己的人(或团队)的一个详细介绍，和使用经验总结。

      我觉得愿意写出这种东西的人，都是伟大的，他们愿意虚心接受，他们不吝分享。致敬！

     

    + - 1. 简介

    • + 对Scrum进行的尝试: 多种团队尺寸(3-12人)，多种sprint长度(2-6周)，多种不同定义的“完成”的方式，多种形式的产品Backlog和Sprint-backlog，多种策略等等。

      + 模型中各部分的解释：
      - Product Backlog (产品需求列表)：由 Product Owner 负责；
      - Sprint Backlog (每次冲刺的需求列表)；
      - Sprint (冲刺)：也就是说要把每次的 Sprint-Backlog 内的开发内容以最快的速度完成；
      - Scrum 中的主要会议有：1. Sprint Planning Meeting(计划会议); 2. Daily Scrum Meeting (每日会议：汇报昨天完成了什么，今天要完成什么--重要工具：Sprint burn down 燃尽图)；3. Sprint Review Meeting (最终的评审会议)，还可以再加一个 Sprint Retrospective Meeting(总结回顾会议)。

     

    + - 2. 我们怎样编写产品BACKLOG

    • + 产品Backlog，最好要按照重要性级别对里面的各项进行排序。
      可参考的针对产品Backlog的模式(Schema)：
      ProductBacklog(ID, Name, Importance, Initial-estimate(人天), How-to-demo, Notes)

      + 为了实现其他之目的，我们有时也可为 ProductBacklog 添加额外的冗余故事字段！
      + 当产品的 ProductBacklog 中涉及到技术层面的术语，那么还需要真正负责技术的人员多问问 “Why” 以确保自己知道客户的真实意图！

     

    + - 3. 我们怎样准备SPRINT计划

    • 切记：在 Sprint-Planning-Meeting(计划会议) 之前，一定要先确保产品的 Product-Backlog 已经井然有序！

       

      + 什么样的 Product-Backlog 叫 “井然有序”？
      1. 它必须存在；
      2. 只能有一个Product-Backlog和一个Product-Owner;
      3. 所有重要的backlog条目都要按照其重要程度评好分；
      4. Product-Owner本人要理解每个故事的含义(他可能不需要知道每个故事要怎么实现，但他必须知道为什么这个故事会出现在这里)！

     

    + - 4. 我们怎样制定SPRINT计划

    • 切记：Sprint-Planning-Meeting(计划会议) 在整个Sprint中至关重要！可以说是此次Sprint中的万物之母(根基)！

       

      + Sprint Planning Meeting(计划会议) 的 实际产出物：Sprint 目标；团队成员名单；Sprint Backlog; Sprint 演示日期；确定每日scrum会议的时间地点等。

      + 为什么产品负责人必须参加Planning-Meeting?
      因为在这个会议上，开发团队和负责人进行面对面交流，会帮助整个团队在对该产品的(Scope, Estimate, Importance)这三者之间做出最优的调整优化！

      + 为什么不能在质量上让步？
      经验告诉我们：牺牲内部质量是一个糟糕透顶的想法！现在节省下来的时间，在接下来的日子里，你一定会一直为它付出更多的代价！
      @联系：自己开发网站，由于刚开始怎么方便怎么来，所以写出来的代码中有大量重复代码，另外网站的整体架构也不是很清晰，数据库也没有进行很好的设计，这就是典型的“内部质量”不过关！这个问题直接导致我后面再有什么新的idea想要添加进入我的网站，我就得花大量的精力把之前的代码也得调整一遍，真的是费时费力！后来自己不得不重花精力好好的为我的代码做一下设计，费了很大力气！

      + 如何处理“无休止的会议”问题？
      一个好的建议就是("时间盒管理法")：会议开始前就事先定好一个会议时长，一旦时间到了会议还没结束，也要立即打断会议，在后续根据实际情况可以对时间盒长度做出恰当调整！

       

      + Sprint-Planning-Meeting 中需要考虑的问题及其建议：
      1. 如何确定会议日程(会议安排)；
      2. 如何确定sprint长度(要自己主动去尝试去实践，作者的实践结果是3周)；
      3. 如何确定sprint目标(能够让大家知道自己这段时间在干什么，这个目标要用业务术语表达，而不是使用技术词汇，以便让团队内外的人都能够理解)；
      4. 如何确定sprint Backlog(4.1从产品负责人角度考虑；4.2从开发团队角度考虑)！

      + 如何确定Sprint Backlog，如何对开发团队的生产率进行合理的估算(三种方法)？
      a. 直觉反应；
      b. 基于昨日天气的生产率计算；
      c. 基于 人天数 和 估计投入程度 的生产率计算！
      通常几种方法会联合起来一起使用，并结合实际做出调整，从而确定最佳的故事点个数，进而帮助确定 Sprint-Backlog。

      建议(Tips)：在实际会议讨论过程中，最好引入“索引卡/小卡片”，这样就可以让更多参与讨论的人有更大的参与感，而且整个讨论过程也显得更加直观。

       

      + Sprint-Planning-Meeting 中需要考虑的问题及其建议(续)：
      5. 如何定义“完成”(多使用一些日常的认识)；
      6. 如何做“时间估算/故事点估算”(要求团队里的每个成员都要相互独立的对故事进行估算，然后再对比对估算结果进行调整)；
      7. 如何明确故事内容,让这个产品团队对其中的故事不会产生歧义(确保每个故事的所有字段都被填满)；
      8. 把故事拆分开来(大故事拆分成小故事，把故事拆分成任务)。
      9. 一定要注意：定下“后续Daily-Meeting的时间和地点”；
      10. 如何处理 “技术故事”问题(试着将技术故事变成普通故事，或是将其变成另一个故事中的某个任务，或者是直接挑明就当做一个技术故事)。

      @关于测试驱动开发(TDD)的实践：针对团队想要开发的每一个故事，第一个任务就是“编写一个失败的测试”，然后想办法实现这个测试所需要的功能，最后一个任务是“对代码进行重构(提高代码可读性并进行代码消重)”。

       

      + 总结：如果整个团队里的每个人在离开会场时都面带微笑，第二天醒来时也是面带微笑，在每一次的每日例会上也面带微笑，那么无疑你的 Sprint-Planning-Meeting 就开的很成功！
      {因为每个人对自己要做什么，怎么做，什么时候做完都已经胸有成竹，所以他们工作起来就会很充实很开心。}

    + - 6. 我们怎样编写 SPRINT-BACKLOG

    • + 最有效的管理 sprint-backlog 的方式：挂在墙上的任务板！
      一个典型的 sprint-backlog 任务板：

      + CHECKED OUT: 代表这个任务被别人认领了，DONE: 代表这个这个任务已经完成了，BURNDOWN(燃尽图) 代表当前完成情况与预期情况的对比。

       

      + 可以利用任务板及时了解到整个团队的进度，并且可以一目了然的看到整个项目的发展状况，可据此给开发团队提供建议！

     

    + - 7. 我们怎样布置团队房间

    • + 大多数最有趣最有价值的设计讨论，都是在任务板前面自然而然地发生的。

      + 设计团队房间的一条重要原则：“要让团队坐在一起”！另外也可让 Product-Owner(产品负责人) 以及 项目经理 等人适当的远离团队，以保证团队可以“凝结”为整体，这些人只需要偶尔去看看这个团队的情况并为团队的进展提供辅助帮助就行了。

     

    + - 8. 我们怎样进行每日例会

    • + 每日例会(Daily-Meeting)：更新任务板(切记要尽力让整个团队参与到保持sprint-backlog及时更新的工作中来) -> 例会结束后需要有人算出剩余工作的时间估算之和，以便更新 sprint燃尽图，让整个团队能更直观地了解项目进度！

      + 如何机智的处理 开会迟到 问题？
      见仁见智，他们的做法是“放一个储蓄罐，迟到的不问缘由往里面放入定额的钱”。

      + 如何处理 那些“不知道自己今天该干些什么”的人？
      a. 重新帮他们梳理任务，让他们自己找出要做的任务；b. 结对编程； c. 用sprint目标 来测试这个团队是否真的无事可做，从而为他们找出事情去做； d. 如果这些不知道干些什么的人还是无事可干，就可以来点狠的了(羞辱式做法、守旧式做法、施加压力法、奴役式做法)，如果这些方法还是对他无用，基本就可以考虑把他踢出团队了！

     

    + - 9. 我们怎样进行SPRINT演示

    • + 我们为什么要进行演示？
      a. 团队的工作成果可得到认可，他们会感觉很好；
      b. 促进团队之间的交流；
      c. 可以迫使团队做出一些真正有意义的工作。
      @联系：我们每周的讨论，这个东西虽然你我感到很烦，但是也正是因为有这个讨论的存在，才迫使我每周都不得不梳理一下我的工作，每周都完成一些工作！因为如果你不完成，你讨论的时候就会变得无话可说！

      + 怎么做演示？
      a. 首先一定要确保清晰阐述自己的目标是什么，你到底想要干什么！
      b. 不要花太多时间来准备演示，这样反而会得不偿失；
      c. 演示要注意将注意力放在“我们做了什么”上面，而不是强调“我们怎么做的”；
      d. 不要去讲一些与目标无关的一大堆细碎和微不足道的东西，这样会容易让听众分散注意力，使得听众找不到真正重要的东西！

      + 怎样处理所谓的“无法演示”或“不好演示”的工作？
      就把你在做这个工作的过程中是怎么评判的讲出来就行了，因为“聊胜于无”。

     

    + - 10. 我们怎样做SPRINT回顾

    • + 为什么团队在结束一个sprint之后要进行回顾？
      - 因为如果没有回顾，你就会发现团队在不断地重犯同样的错误！回顾是对前一段工作进行总结和评价的最佳时刻！
      - 我们每一次进行回顾的潜在主题都是：“我们怎样才能够在下个sprint过程中做的更好”！

      + 进行总结和回顾，它的实际输出是什么？
      - 它的输出是 <Good, Could-have-done-better, Improvements> 这三项组成的一系列想法列表！
      a. Good 和 Could-have-done-better 是用于回顾过去的，看看过去的工作中有哪些做法是做的很好的(继续保持)，有哪些做法是本可以做的更好的；
      b. Improvements 是用于展望未来的，根据前面的总结我们看看在将来的工作中，我们可以做出什么样的改进，以保证不重犯同样的错误！
      切记: 不要试图一口气吃成个胖子，每个sprint只应该关注几个改进就够了，不要做太多。

      + 建议：把sprint回顾结果贴到团队房间的墙上 往往会更有效！

      + 回顾会议中发现的常见问题及其处理方法：
      Q1: 我们应花更多时间，把故事拆分成更小的条目和任务；
      Q2: 有太多的外界干扰；
      Q3: 我们做出了过度的承诺，最后只完成了一半工作；
      Q4: 我们办公室的环境太吵太混乱了...

     

    + - 11. SPRINTS之间的修整时刻

    • + 切记：各个sprint之间一定要有适当的休息时间！
      就像一个上紧了的发条一样，总是绷得那么紧，始终高速工作，实际上收到的成效反而不好！

      + 总之，在下一个sprint计划会议之前，每个人应该至少度过一个不需要考虑sprint的夜晚！[比如：sprint1总结安排在周五，sprint2计划会议安排在下周一]。
      + “实验日 (Lab Day)” 是另一种休息的方式，在这样的日子里，开发人员基本可以做任何他自己想做的事情，从而让大家能得到自然的休息，同时也可能为整个开发团队带来意想不到的收获。

     

    + - 12. 我们怎样制定发布计划，处理固定价格的合同

    • + 在实际项目中，你会发现大多数时候一次只计划一个sprint是明显不足的，你需要提前多做些计划！
      多做计划，预先做计划，能够防止我们遇到无法交付的危险。你要时常问自己：“最晚到什么时候，我们可以交付这个新系统的1.0版本？”

      + 考虑制定合适的验收标准  (可以根据每个story的重要程度来定义)；
      + 对重要条目进行时间估算；
      + 对团队生产率进行估算，从而得到预计可完成的故事个数。

       

      + 有了上述的验收标准、时间估算、团队生产率估算之后，我们就可以很容易的将 Product-Backlog 拆分到各个 sprints 中去，同时也就得出了一个理想状态下的发布时间，通常还需要再附加一个“缓冲期”，以防计划中出现意想不到的情况。

      + 如果在产品开发过程中，真的出现了什么意外情况，我们就可以考虑 [找出原因，看能否内部解决，如果不能，就需要调整发布计划]！

     

    + - 13. 我们怎样组合使用Scrum和XP

    • + 组合使用 Scrum和XP(极限编程) 往往能大大提高团队生产率！

       

      + Scrum 注重的是管理和组织实践，XP 关注的是实际的编程实践。
      + XP 实践：
      a. 结对编程；
      b. 测试驱动开发 (TDD)；
      测试驱动开发意味着你要先写一个自动测试，然后编写恰好够用的代码，让它通过这个测试，接着对代码进行重构，主要是提高它的可读性和消除重复。整理一下，然后继续。
      c. 增量设计；
      d. 持续集成；
      e. 统一代码标准；
      (- 默 认 使 用 Sun 的 代 码 惯 例 ：http://java.sun.com/docs/codeconv/html/CodeConvTOC.doc.html)
      (- 需要返回 Collections 或者数组的方法不应该返回 null。应该返回空的容器或数组，而不是 null。)
      f. 保证可持续的开发速度以及团队成员精力充沛的工作！

     

    + - 14. 我们怎样做测试

    • + 测试往往都是做软件过程中最困难的部分

       

      + “验收测试阶段”：为了让你的终端用户不会面临问题重重的1.0.0版本，团队必须发布一个经过了“验收测试阶段”的1.0.1版本！
      这个阶段：是指整个测试、调试、重新发布阶段，直到得到可以用来做产品发布的版本为止！

      + 既然我们无法避免验收测试，那么怎么做可以让我们把验收测试阶段缩到最短？
      a. 全力提高 Scrum 团队交付的代码质量；
      (即，1. 把测试人员放到 Scrum 团队中；2. 每个 sprint 少做点工作)
      b. 全力提高人工测试工作的效率。
      (即，1. 找到最好的测试人员；2. 给他们最好的工具；3. 确保他们上报那些耗费时间、却能够被自动化完成的工作)

      + 实践经验：要全力提高 Scrum 团队发布的代码质量，在一个 sprint 中及早发现并修复 bug，这要比 sprint 结束以后再这样做的代价小得多。

       

      + 在上一个sprint结束后，旧版本可以产品化之前，我们到底要不要构建新特性？
      比较好的处理方式是：可以开始构建新东西，但是要给‘将旧功能产品化’分配高优先级！
      最差的处理方式是：只关注构建新东西，而不想要去把旧版本产品化。这往往会让你得不偿失！

     

    + - 15. 我们怎样管理多个SCRUM团队

    • 思考：当一个产品本身很复杂，涉及到很多人同时对其进行开发时，请思考：“我们到底应该创建多少个团队？”，“我们到底要如何把这些人员分配到各个团队？”

       

      + 到底要不要把所有人放在同一个团队里？
      大团队小团队各有优劣点，大团队人多办事效率会降低，小团队人少但是一旦拆分开来，就必须确保他们彼此之间不会产生互相干扰(要相互独立)，否则沟通成本就会上升，团队办事效率也会降低。
      试想：甚至是每天的 Daily-Scrum 会议基本上都会超过15 分钟，每个人都不太清楚其他人在做什么，所以整个状态就会混乱。关于沟通成本上升导致办事效率降低，请联想《人月神话》里面的经典解读。
      总之，针对这个问题：最好根据自己的实际情况，自己去实践和改进，找出最适合自己的决定！

      + 经常观察团队内的所谓的“虚拟团队(virtual team)”，这可以为你分配和管理这些团队提供参考意见！

      + 如果同一个产品有多个团队，那么建议你可以考虑“将多个sprint进行同步”！

      + 关于如何决定各个团队内的人物角色，大家如何分组，以及各种跨团队问题，我们决定选取一个专门的“团队领导”这个角色，来负责这个任务！时间证明这种方法还挺好！

      + 如何为多个团队分配人手？
      最开始使用一些集中式控制，然后再用分散式对其进行优化！

      + 要不要组建负责特定功能的专业团队？
      实践证明，既然分了团队，就一定要保证团队之间的沟通成本要降到最低，因此要保证尽量让各个团队之间都相互独立，互不影响！所以，从这个角度来看，我么不鼓励使用所谓的“专业团队”，而应该使用自己能够独当一面的“综合团队”！
      不过，要是有很强烈的需求，我们也会临时创建针对特定组件展开工作的专业团队。

      + 要不要频繁重组各个团队？
      不到必须出手的时候，最好不要乱重组，因为“团队凝聚力”是Scrum的核心要素之一，如果一个团队合作工作达多个sprint之久，他们就会变得非常紧密。团队生产力会提升至难以置信的地步。不过要达到这个地步需要花上一定时间。如果不断变换团队组成，你就永远无法得到强悍的团队凝聚力。

      + 什么是所谓的 Scrum-of-scrums?
      Scrum-of-scrums 实际上是一个常规会议，是为了让所有的 Scrum master 能够有机会聚到一起进行交流。这是个发布概要信息的论坛，而不是提供讨论或者反映问题的场所。

      + 对多团队之间的协同工作的其他经验与意见：
      a. 使用交错的每日例会(Daily-Meeting);
      b. 在有必要的情况下，可以考虑设立单独的“救火scrum团队”；
      c. 即使是多团队，也还是建议你采用“一个产品负责人，一个产品backlog”的策略来对他们进行管理，虽然过程会混乱，但是还是比其他方式更合理有效的！

     

    + - 16. 我们怎样管理地理位置上分布的团队

    • + Scrum和XP的大部分魔力都来自于团队成员们最好都能身处同地紧密协作，可以结对编程，可以做到每日面对面交流。
      所以如果有一个团队地理位置是分散的，那么我们也只好“想尽办法来把物理位置上分散的团队成员之间的沟通带宽增至最大”！

      + 总之，处理的策略就是：观察->调整->观察->调整->... ...
      你可以采用你自己觉得不错的方式，然后实验，不行就调整，最终你会发现一个适用于你们的策略。而针对这个问题，我们的策略是设立一个“聚焦日”。
